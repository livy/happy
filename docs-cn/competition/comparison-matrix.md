| 供应商 | 核心传输 | 文本形状 | 子代理 / 任务 | 权限 | 沙箱情况 | 同步 / 远程情况 | Happy 启示 |
| --- | --- | --- | --- | --- | --- | --- | --- |

| Claude | ACP JSON-RPC 加本地文件系统状态 | 流式会话更新 + 本地文本文件 | 代理团队、收件箱、任务、子代理 JSONL | 模式 + 权限是一流的，但分散在 ACP 和本地设置中 | 更窄的 shell 沙箱加上信任 / 权限层 | 本地优先，存在远程控制桥，丰富状态仍泄漏到磁盘 | 最佳代理工作流参考，作为单一干净协议较弱 |
| Superset | Hono HTTP + tRPC + WebSocket EventBus | 不透明 — 不解析代理输出，通过钩子观察生命周期 | 在真实 PTY + 每个任务的 git 工作树隔离中与代理无关的启动 | 不适用 — 委托给每个代理自己的权限模型 | 工作树隔离，代理在真实终端中运行并带有自己的沙箱 | Electric SQL 云→本地同步 + 用于 CLI→桌面控制的云 DB 命令队列 + 基于清单的主机服务持久性 | 最佳编排层参考；最强的包边界；Electric SQL 同步和云命令队列模式值得研究 |

## 当前解读

- OpenCode 是目前 Happy 最具吸引力的端到端参考。
- Codex 拥有最清晰的类型化应用服务器模型，用于线程、回合、项目、审批和沙箱策略。
- Claude 拥有最成熟的代理团队工作流，但其有用状态分散在 ACP、钩子、变更日志行为和 `~/.claude/` 中。
- Superset 是编排层设计的最强参考 — 不拥有代理协议，只是协调。卓越的发布速度（3 人，5 个月内 2,100+ 次提交）。

## Happy 的建议设计方向

- 使用 OpenCode 的信封 + 类型化部件文本模型作为主要 UI/会话协议参考。
- 借鉴 Codex 的显式服务器请求模式用于审批和用户输入。
- 保留 Claude 的代理团队经验，但避免依赖隐藏的本地文件作为主要事实来源。
- 将待办事项、权限、问题和子代理视为一流的状态通道，而不是助手文本的 hack。
- 接下来研究 OpenCode 的服务器同步路径；这看起来像是最高杠杆的后续行动。
- 评估 Superset 的 Electric SQL 云→本地同步模式和云 DB 命令队列，作为 Happy 当前同步方法的替代方案。
- 考虑为 Happy 的 CLI/服务器拆分使用 Superset 的主机服务提取模式（可注入提供者、基于清单的持久性）。
